# 页面与角色视图工作台规范

更新时间：2026-04-27

这份文档用于后续新项目、新增页面或沉淀 Codex skill 时复用。目标是把“页面交付物”做成产品、设计、前端、后端、算法、测试、管理评审都能使用的统一工作台。

## 1. 总体原则

工作台不是单纯展示页面说明，而是把一个项目从项目级背景到页面级交付块完整组织起来。

核心原则：
- 项目先于页面：先有项目总览，再进入具体页面。
- 页面是主对象：每个业务页面都有独立交付块。
- 交付块标准化：摘要、模块、设计稿、接口、资源、字段、规则、状态、验收、风险都结构化。
- 角色视图组合展示：不同角色看到同一页面的不同交付块组合。
- 原始资料要保留：Mock Markdown、设计稿、接口平台都要可打开或嵌入。
- 外层必须可视化：不要只堆长文本，外层要让人一眼看懂。
- 数据要可导出：字段、接口、Mock 相关内容尽量支持复制 Markdown、下载 CSV/Excel。

## 2. 项目级总览

每个项目必须有一个独立的“项目总览”Tab，放在页面 Tab 上方。

项目总览至少包含：
- 项目介绍：项目目标、业务背景、服务对象、核心价值。
- 项目级用户流程：从进入平台到浏览页面、引用模板、进入工作区、评审联调测试的主链路。
- 页面地图：当前页面、后续扩展页面。
- 核心业务对象：模板、项目、设备、指标、Mock、字段、接口、设计稿。
- 产品范围：本期做、本期不做、哪些前端写死、哪些接口固定 JSON。
- 技术关注点：静态前端、配置驱动、角色视图、本地存储、iframe 嵌入等。
- 接口与数据策略：Mock 来源、字段定义、接口契约、数据兜底。
- 质量与验收：页面级、数据级、交互级验收。
- 风险与依赖：iframe 限制、外链失效、指标口径、枚举变化、Mock 与接口不一致。

项目总览不受角色视图筛选影响。

## 3. 页面 Tab

页面 Tab 只放具体业务页面，例如：
- 探索页
- 工作区页面
- 规划页
- 数据集页
- 项目详情页

不要把项目总览混在页面 Tab 内。项目总览是项目级入口，页面 Tab 是页面级入口。

每个页面必须包含：
- 页面名称
- 页面副标题
- 页面摘要
- 模块数量
- 接口关注度
- 风险项数量
- 页面 blocks
- 角色 views

## 4. 页面摘要

页面摘要不要做成普通长列表，应该做成可视化摘要卡。

推荐摘要卡：
- 页面定位
- 核心目标
- 本期范围
- 非本期范围

示例：
- 核心目标：展示标杆项目，供给模板。
- 非本期范围：外部情报自动抓取。
- 实现口径：本期前端写死，或接口固定 JSON。

页面摘要必须服务“快速理解页面”，不要写成需求长文。

## 5. 页面内分区

页面内容按以下分区组织：

1. 页面摘要
2. 当前角色关注重点
3. 待开发页面模块 PRD
4. 设计稿
5. 接口
6. 资源
7. 折叠详情区

注意：
- “待开发页面模块 PRD”需要高亮标注，因为它是研发拆任务的核心区域。
- “设计稿 / 接口 / 资源”不要放在一起，必须拆成三个独立板块。
- 页面没有的模块不要放占位。例如工作区没有信息区，就不要出现信息区模块和信息区原型图。

## 6. 待开发页面模块 PRD

模块是页面要开发的功能区域，不是资源。每个模块卡应同时包含“简洁版 PRD”和“详细版 PRD”。

简洁版 PRD：
- 用短 item 罗列当前模块的主要功能。
- 适合管理评审、快速扫读、页面概览。

详细版 PRD：
- 按模块目标、用户故事、功能边界、主流程、关键验收点展开。
- 适合产品、研发、测试快速理解模块意图和范围。
- 不重复字段定义、接口路径、Mock 明细和完整状态全集；这些内容由字段、接口、资源、状态与交互板块承载。
- 默认可折叠，避免页面过长。

模块命名建议：
- A. Header 模块
- B. 案例卡片区模块
- C. 信息区模块

模块内容必须拆成单独 item，不要用“+”串联多个功能。

错误表达：
- 文字标签 + 电站图片 + 项目标题 + 一句话简介。

正确表达：
- 展示文字标签。
- 展示电站图片。
- 展示项目标题。
- 展示一句话项目简述。

如果某页面没有某模块，就不要出现该模块。

## 7. 设计稿板块

设计稿板块用于展示原型图、卡片图、信息区图等。

每个设计稿卡片应包含：
- 标题
- 预览图
- 简短说明
- 导航到设计稿
- 打开原图 / 原型
- 点击图片放大预览

如果存在 Figma、蓝湖、摹客、Apifox Design 等设计平台链接，使用 `designUrl` 做“导航到设计稿”。

如果没有单独设计平台链接，可以临时使用图片链接作为导航目标，但后续应替换。

## 8. 接口板块

接口模板不要只是“待补充”文本，应做成接口卡。

接口卡应包含：
- 接口状态：待定义 / 已定义 / 已联调 / 已冻结。
- 负责人。
- 更新时间。
- Swagger / Apifox / OpenAPI JSON 链接。
- 页面内 iframe 嵌入预览。
- 外部打开兜底。
- 接口清单表。

接口清单字段：
- Method
- Path
- 接口名称
- 说明
- 状态

iframe 嵌入策略：
- 如果 Swagger / Apifox 允许 iframe，则页面内直接展示。
- 如果被 `X-Frame-Options` 或 `Content-Security-Policy` 阻止，保留外部打开。
- 私有 Apifox 可能受登录态和跨站 Cookie 影响，必须有兜底。

没有真实链接时，应展示嵌入占位页和接口清单，而不是空白。

## 9. 资源板块

资源板块放 Mock 数据、Markdown 原文、配置样例等。

Mock 数据不能只放普通列表。应做成两层：
- 外层：美化后的可视化摘要，让大家一眼看懂。
- 内层：完整 Markdown 原文，可弹窗查看。

Mock 数据卡应包含：
- 数据来源。
- 模板数量或案例数量。
- 页面布局摘要。
- 核心案例卡片或模板组件流。
- 查看完整 Markdown。

Mock 原始 Markdown 必须完整保留。

## 10. 字段定义

字段定义必须做成可视化表格，不能只写 `field(type)：说明`。

字段表列：
- 所属区域
- 字段名
- 中文名
- 类型
- 单位
- 必填
- 来源
- 示例
- 展示规则
- 校验规则
- 兜底
- 操作

字段必须按页面区域分组，例如：
- A. Header 字段
- B. 案例卡片字段
- C. 信息区字段

字段表交互：
- 每个字段单独复制 Markdown。
- 每个字段单独下载 CSV。
- 每个字段单独下载 Excel。
- 每个分组整体复制 Markdown。
- 每个分组整体下载 CSV。
- 每个分组整体下载 Excel。

单位规则：
- 文本字段：无。
- 图片 URL：无。
- 枚举：无。
- 数组：无。
- 装机量：MW。
- 百分比：%。
- 成本：元/kWh、元/kg。
- 收益：万元/年。
- 碳减排：万吨/年。
- 时间：ISO 8601、小时/年等。

指标字段建议拆开：
- `highlight_metric_key`
- `highlight_metric_label`
- `highlight_metric_value`
- `highlight_metric_unit`

亮点指标必须以当前 Mock 数据为准，不要臆造固定三选一。

## 11. 角色视图

角色视图用于不同岗位按需看页面交付块。

建议角色：
- 全量视图
- 产品视图
- UI / UX 视图
- 前端视图
- 后端视图
- 算法视图
- 测试视图
- 管理 / 评审视图
- 自定义视图

默认刷新后进入“项目总览”。进入页面后默认角色为“全量视图”。

“完整视图”这个名字有歧义，应使用“全量视图”。

## 13. 测试视图重点

测试视图要能直接拆测试用例。

必须包含：
- 页面摘要。
- 用户流程。
- 页面结构。
- 页面范围。
- 待开发页面模块。
- 设计稿。
- 接口。
- Mock 数据。
- 字段定义。
- 业务规则。
- 状态与交互。
- 验收标准。
- 风险。

重点测试项：
- 加载态。
- 空态。
- 错误态。
- 无权限态。
- 搜索无结果。
- 图片预览。
- Markdown 弹窗。
- 字段复制/下载。
- iframe 嵌入失败兜底。
- 外链打开。
- localStorage 自定义视图。

## 14. 新增页面流程

新增一个页面时按以下顺序执行：

1. 明确页面名称、副标题、页面位置。
2. 写页面摘要卡：页面定位、核心目标、本期范围、非本期范围。
3. 写关注重点：目标、范围、结构、流程。
4. 拆待开发页面模块 PRD，按 A/B/C 命名，并同时写简洁版和详细版。
5. 挂设计稿：整体图、模块图、必要的局部图。
6. 写接口卡：Swagger/Apifox/OpenAPI 链接、嵌入、接口清单。
7. 写资源卡：Mock 可视化摘要和完整 Markdown。
8. 写字段定义表：按区域分组，补齐单位、示例、校验、兜底。
9. 写业务规则。
10. 写状态与交互。
11. 写验收标准。
12. 写风险和依赖。
13. 配置各角色视图。
14. 检查不存在的模块不要出现占位。
15. 跑语法检查。

## 15. UI 规范

UI 应该像工作台，不像营销页。

要求：
- 左侧为项目、页面、角色、页内导航。
- 侧边栏支持收起。
- 主内容保持信息密度，避免过大留白。
- 卡片圆角保持 8px 左右。
- 页面模块区需要高亮。
- 表格支持横向滚动。
- 外链和嵌入都要有兜底。
- 文本不要溢出按钮或卡片。
- 不使用装饰性渐变球、花哨背景。

## 16. 后续 Skill 方向

后续可以把这份文档沉淀成一个 Codex skill，触发场景：
- 用户要做新项目的产品工作台。
- 用户要新增一个页面。
- 用户要把需求文档转成页面交付块。
- 用户要为页面生成角色视图。
- 用户要生成字段表、Mock 可视化、接口卡。

Skill 应该读取输入材料，然后输出：
- 项目总览。
- 页面配置。
- 角色视图配置。
- 字段表。
- Mock 数据可视化。
- 接口卡。
- 静态前端页面或数据配置。
